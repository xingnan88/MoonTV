# MoonTV 邀请码登录 PRD

## 1. 目标

将 MoonTV 普通用户登录改为 **6 位纯数字邀请码**。

- 每个人一个不同邀请码。
- 邀请码绑定内部 `username`，用于区分观看记录、收藏、搜索历史和跳过配置。
- 邀请码支持三种有效期：一周、一个月、一年。
- 管理员可创建、续期、禁用、启用、删除邀请码。
- 邀请码过期、禁用或删除后，已有会话也必须失效。
- 当前站长 `admin` 保持独立管理员登录能力。

不做：

- 不接入 Cloudflare KV。
- 不支持用户自助注册。
- 不支持自定义有效期。
- 不迁移现有播放记录和收藏数据。

## 2. 角色与登录

| 角色     | 登录方式                                                | 权限    |
| -------- | ------------------------------------------------------- | ------- |
| 普通用户 | 输入 6 位数字邀请码                                     | `user`  |
| 站长     | 独立管理员入口，输入 Cloudflare `USERNAME` / `PASSWORD` | `owner` |

普通用户不知道内部用户名。服务端通过邀请码查到 `username` 后，继续用该 `username` 保存用户数据。

## 3. 核心流程

普通用户登录：

1. 打开登录页。
2. 输入 6 位数字邀请码。
3. 服务端校验邀请码格式、启用状态和有效期。
4. 校验通过后生成登录 Cookie。
5. Cookie 中身份为邀请码绑定的内部 `username`。

管理员创建邀请码：

1. 进入管理后台。
2. 选择有效期：周、月、年。
3. 系统生成内部用户名和 6 位数字邀请码。
4. 管理员复制邀请码发给用户。

续期：

- 从 `max(当前时间, 原失效时间)` 开始延长。
- 原邀请码不变。

失效：

- 邀请码到期、禁用或删除后，用户不能登录。
- 已登录会话访问受保护接口时也必须失效。

## 4. 邀请码规则

- 格式：6 位纯数字，例如 `583729`。
- 允许前导零，例如 `003821`。
- 服务端必须校验：`^\d{6}$`。
- 邀请码由服务端生成，不允许手工指定。
- 使用安全随机数生成。
- 邀请码必须唯一。
- 登录失败提示统一为“邀请码无效或已失效”，避免枚举。
- 登录接口必须做频率限制。

## 5. 数据设计

复用现有 D1 `users` 表：

```sql
ALTER TABLE users ADD COLUMN invite_expires_at INTEGER;
ALTER TABLE users ADD COLUMN invite_enabled INTEGER NOT NULL DEFAULT 1;

CREATE UNIQUE INDEX IF NOT EXISTS idx_users_invite_code
ON users(password);

CREATE INDEX IF NOT EXISTS idx_users_invite_expires_at
ON users(invite_expires_at);
```

字段含义：

| 字段                | 含义                         |
| ------------------- | ---------------------------- |
| `username`          | 内部用户 ID，用于隔离数据    |
| `password`          | 邀请码                       |
| `invite_expires_at` | 邀请码失效时间，Unix 时间戳  |
| `invite_enabled`    | 是否启用，`1` 启用，`0` 禁用 |

内部用户名由系统生成，例如 `user_x8k29a`，不展示给普通用户。

## 6. API

普通邀请码登录：

```http
POST /api/login
```

```json
{
  "inviteCode": "583729"
}
```

管理接口：

| 方法     | 路径                 | 用途               |
| -------- | -------------------- | ------------------ |
| `GET`    | `/api/admin/invites` | 邀请码列表         |
| `POST`   | `/api/admin/invites` | 创建邀请码         |
| `PATCH`  | `/api/admin/invites` | 续期、禁用、启用   |
| `DELETE` | `/api/admin/invites` | 删除邀请码对应用户 |

受保护接口要求：

- 普通用户请求时必须检查用户仍存在。
- 必须检查 `invite_enabled = 1`。
- 必须检查 `invite_expires_at > now`。
- 不满足时返回 `401` 并清理登录状态。

## 7. 页面要求

登录页：

- 普通登录只显示一个 6 位数字邀请码输入框。
- 输入框只允许数字，最多 6 位。
- 提供独立管理员登录入口。
- 非 6 位数字不能提交。

管理后台：

- 可创建周、月、年邀请码。
- 可查看邀请码、状态、创建时间、失效时间。
- 可复制邀请码。
- 可续期、禁用、启用、删除。
- 删除时提示是否同时删除播放记录、收藏、搜索历史和跳过配置。

状态：

| 状态     | 条件                     |
| -------- | ------------------------ |
| 有效     | 已启用且未过期           |
| 即将过期 | 有效且剩余不超过 3 天    |
| 已过期   | 失效时间小于等于当前时间 |
| 已禁用   | `invite_enabled = 0`     |

## 8. 验收标准

- 两个不同邀请码能登录为两个不同用户。
- 两个用户的播放记录、收藏、搜索历史互不影响。
- 非 6 位数字邀请码无法提交。
- 无效、过期、禁用邀请码无法登录。
- 禁用或删除邀请码后，已有会话不能继续访问受保护接口。
- 周、月、年有效期计算正确。
- 续期逻辑正确。
- 普通用户不能访问管理后台。
- 站长仍可用管理员入口登录并管理邀请码。
- 登录接口有频率限制。

## 9. Codex 测试要求

实现完成后，Codex 必须：

1. 运行单元测试和接口测试。
2. 启动本地服务。
3. 使用浏览器完成端到端验收。
4. 检查浏览器控制台无相关错误。
5. 保存登录页、邀请码管理页、普通用户登录后页面截图。
6. 输出测试命令、通过项、失败项、截图路径。

浏览器必测流程：

1. 登录页只显示邀请码输入框和管理员入口。
2. 输入非法邀请码时不能提交。
3. 管理员登录后台。
4. 创建周、月、年邀请码。
5. 用邀请码登录普通用户。
6. 普通用户无法进入管理后台。
7. 两个邀请码的数据隔离。
8. 禁用邀请码后刷新页面，会话失效。
